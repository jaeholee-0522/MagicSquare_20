## 4. Golden Master 아키텍처

### 4.1 캡처 전략

| 방식 | 채택 | 비고 |
|---|---|---|
| stdout capture | — | ECB GUI/CLI 분리; 미채택 |
| **API result serialization** | ✅ | `SolveUseCase.execute(grid)` → DTO 직렬화 |

성공: `Output:\n[r1, c1, n1, r2, c2, n2]`  
Boundary 실패: `Error:\n<ValidationErrorCode>`  
Domain 실패: `Error:\n<DomainErrorCode>`

### 4.2 Approve 패턴

| 조건 | 동작 |
|---|---|
| baseline **없음** | 현재 출력으로 자동 생성 (`created`) |
| baseline **일치** | PASS (`matched`) |
| baseline **불일치** | `--- expected` / `+++ actual` unified diff → FAIL |
| `--approve-golden-master` | baseline 덮어쓰기 (`approved`) |

구현: `tests/golden_master/approval.py` — `approve_snapshot()`, `approve_section()`, `read_expected()`

### 4.3 파일 구조

```text
tests/
├── golden_master_expected.txt          # version-controlled baseline
├── test_golden_master_magic_square.py  # GM-2 pytest entry
└── golden_master/
    ├── scenarios.py                    # GM-TC-01~05 + serialization
    ├── approval.py                     # approve / diff
    └── contracts.py                    # int[6], row-major, 1-index, rules

scripts/generate_golden_master.py       # standalone baseline generator
docs/golden_master_approval.md          # design document
```
